ARIA
Accessible Rich Internet Applications

WAI-ARIA - Web Accessibilty Initiative for Accessible Rich Internet Applications

  1. If you can use a native HTML element with the behavior and semantics needed already implemented, instead of repurposing other element adding ARIA roles, states and properties, do it.
  2. Do not change native semantics
  3. All interactive controls ARIA must be usable through the keyboard
  4. All interactive elements must have an accessible name
  5. Not using ARIA is better than using it wrong

When native HTML won't cover the case, we can use ARIA to change and augment the accessibility tree

roles

Shorthand indicator for a particular region or UI pattern

aria-label

Specifies a string to be used as the accessible label, will override any other native labeling mechanism, such as a label element.

aria-labelledby

Specifies through the ID of another element in the DOM, the text to be read as the element's label.

aria-describedby

Specifies through the ID of another element in the DOM, complementary information about the element.

aria-hidden

Excludes content that is not visually hidden from assistive technology

aria-live

Communicates user that a part of the page is updated (live region), regardless of their position on the page. Possible values are off (no announemets), polite (will wait for user to be idle) and assertive (will announce as it changes).

aria-relevant

Indicates the type of changes that should be notified to the user. Possible values are additions, removals, text and all.

aria-atomic

Indicates if a whole region should be communicated when it updates.

aria-busy
Indicates whether changes to an area should be temporarely ignored.
